        추가했음에도 이벤트가 발생하지 않음.


     2. 근본 원인 규명: "앵커(Anchor)와 터치 영역의 분리 결함"
     가장 치명적인 원인은 우리가 마커 아래에 식당 이름(텍스트)을 넣으려고 비트맵의 세로 길이를 늘리고 `Anchor Point`를 변경한 방식에 있습니다.


      * 카카오맵 SDK의 터치 바운딩 박스(Bounding Box) 연산 방식:
         카카오맵 엔진은 비트맵 이미지를 그릴 때 지정한 Anchor Point (0.5f, anchorY)를 기준으로 지리적 좌표(위/경도)에 이미지를 렌더링합니다.
         하지만 터치(Click) 영역을 계산할 때는 사용자가 커스텀하게 만든 동적 비트맵의 가변 크기와 앵커 보정값을 완벽하게 동기화하지 못합니다.
      * 결과적으로, 화면에 마커(빨간 박스 포함)가 그려지기는 하지만 카카오맵 SDK 내부 로직에서는 해당 위치를 "마커가 존재하지 않는 빈 지도(Map)"로
        인식해버리는 현상이 발생합니다. 그래서 마커를 눌러도 무조건 MapClickListener가 터치를 인터셉트(가로채기)해 버리는 것입니다.


     3. 해결을 위한 필수 전략 (Action Plan)
     마커 아래에 텍스트를 커스텀 비트맵으로 무리하게 합쳐 그리는 방식을 폐기하고, 카카오맵 SDK가 공식적으로 지원하는 "텍스트 라벨(Text Label) 조합
     방식"을 사용해야 합니다.


      * 현재 실패한 방식 (폐기 대상):
         하나의 커다란 Canvas에 '캡슐+아이콘'과 '하단 텍스트'를 동시에 그려 통째로 LabelStyle의 비트맵으로 밀어 넣는 방식. (SDK 터치 영역 버그 유발)
      * 새로운 해결 방식 (정석 적용):
          1. MarkerBitmapFactory에서는 다시 순수하게 '캡슐 모양 마커(포크/나이프 + 조회수)' 비트맵만을 반환하도록 롤백합니다. (텍스트 제외, 앵커도
             다시 기본값 1.0f로 복구)
          2. KakaoMapView.kt에서 라벨(마커)을 생성할 때, SDK에서 공식적으로 지원하는 텍스트 결합 기능인 `LabelTextTexts`와 `LabelTextStyle`을
             사용합니다.
          3. LabelOptions.from(..).setTexts("식당이름")을 사용하여 마커 비트맵의 지정된 위치(하단)에 카카오맵 엔진이 직접 텍스트를 렌더링하게
             만듭니다.


     이 방식으로 변경하면 텍스트는 텍스트대로 깔끔하게 보이고, 카카오맵 엔진이 터치 바운딩 박스를 정상적으로 계산할 수 있어 마커 클릭
     이벤트(setOnLabelClickListener)가 완벽하게 복구됩니다.
